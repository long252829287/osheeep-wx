# “小家开饭”正式版产品演进设计

更新日期：2026-07-15

## 1. 背景与目标

当前版本已经完成双人家庭、双方选菜、菜单确认、完成记录、隐私中心和账号注销的 MVP 闭环，但尚未达到正式版标准。本轮正式版演进聚焦五条产品线：

1. 家庭共享食材库存，以及按食材找菜。
2. 系统菜与家庭自定义菜谱、多个做法版本。
3. 可管理的双人家庭。
4. 完整一致的小程序内提示、消息中心和可选微信订阅提醒。
5. 以授权明确的真实菜品照片替换现有生成式图片，并整体提升产品体验。

正式版仍服务两个人共同决定和准备一顿饭，不扩展为公开菜谱社区、多人投票平台或专业仓储系统。

## 2. 已确认的产品原则

- 当前体验继续限定两人家庭，但成员数据模型和接口避免把角色能力写死在控制器或前端判断中。
- 食材库存由家庭共享，记录数量和单位，数量允许留空。
- 找菜时从家庭库存开始，可以临时增加或排除食材；临时条件不直接修改库存。
- 默认展示全部匹配菜谱并按匹配度排序，用户可以开启“只看能做”。
- 每道菜有一个默认做法，也允许家庭成员增加其他做法版本。
- 自定义菜谱先保存为个人草稿，发布后进入家庭共享菜谱。
- 两名家庭成员可以共同编辑已发布菜谱；删除采用归档，历史记录不受影响。
- 系统菜保持只读，可以复制为家庭菜谱后修改。
- 首个正式版不开放图片上传，只允许从审核过的真实照片库选择。
- 菜单完成后再询问是否扣减库存；扣减失败不能回滚或重复生成做饭记录。

## 3. 正式版核心体验

### 3.1 找菜首页

“菜谱”底部导航默认进入找菜首页：

- 显示“今晚想吃什么”和家庭库存摘要。
- 首屏最多显示 3 个当前食材和“展开全部”，避免标签淹没主要内容。
- 提供“调整食材”和“只看能做”。
- 展示一项重点推荐和两项轻量推荐；每项明确显示食材状态、预计耗时和缺少项。
- 只在确有双方偏好依据时显示“你们都喜欢”。
- 提供低干扰的“家庭菜谱”和“食材库存”入口。
- 可以从结果进入详情或直接加入今晚菜单。

选定视觉方向见：

- `docs/design/formal-release/recipe-discovery-final-direction.png`

该图片只用于界面结构和视觉方向，不得把其中的生成式菜品图片作为正式产品资源。

### 3.2 食材库存

- 按肉类、蔬菜、蛋奶、主食、调味料等分类。
- 支持搜索、添加、修改数量和移除。
- 数量使用十进制，单位来自标准单位表；家庭自定义食材可以选择已有单位。
- 数量为空表示“家里有，但数量未知”。
- 双方修改使用版本校验；冲突时保留本地输入并提示重新加载或合并。
- 库存变化记录操作者和时间，但首版不做采购批次、保质期或临期提醒。

### 3.3 匹配规则

每个菜谱食材分为必需和可选。匹配状态按必需食材计算：

- `AVAILABLE`：库存存在，且双方都记录数量时库存不少于菜谱用量。
- `UNKNOWN_QUANTITY`：库存存在，但至少一侧数量未知。
- `MISSING`：库存不存在，或已知库存少于菜谱用量。

排序优先级依次为：无缺失、匹配比例、双方偏好、预计耗时、稳定的菜谱 ID。

“只看能做”排除存在 `MISSING` 的菜谱，但允许 `UNKNOWN_QUANTITY`，并显示“数量待确认”。临时食材条件只在当前找菜会话中生效。

### 3.4 家庭菜谱

家庭菜谱分为“已发布”“我的草稿”“已归档”：

- 草稿只有创建者可见。
- 发布后家庭双方可见、可编辑，并展示创建者和最后修改者。
- 归档后不再参与找菜和今晚选菜；已有菜单和历史快照不变。
- 系统菜可以复制为家庭菜，复制后与系统菜独立演进，但保留来源关联。

创建流程采用分步编辑并自动保存草稿：

1. 基本信息：名称、分类、口味、份量、预计耗时。
2. 所需食材：食材、数量、单位、必需或可选。
3. 默认做法：烹饪方式、有序步骤和备注。
4. 其他做法：可选的“少油版”“空气炸锅版”等。
5. 真实图片：从内部图库搜索和选择。
6. 预览并发布。

共享内容发布或更新前执行服务端文本内容安全检查；草稿保存不在每次输入时调用外部检查。检查失败时保留草稿并明确指出不能发布，不回显平台原始错误。

### 3.5 菜谱详情与今晚菜单

- 菜谱详情展示真实图片、食材、库存满足情况、默认做法和其他做法版本。
- 加入今晚菜单时记录所选做法版本；未选择时使用默认做法。
- 双方选择同一道菜但做法不同，合并菜单需要明确提示并让确认者选择最终做法。
- 菜谱修改后，已经完成的记录继续使用完成时快照。

### 3.6 库存扣减

菜单完成与库存扣减是两个独立、可重试的动作：

1. 原有完成接口幂等生成做饭记录。
2. 客户端进入可跳过的扣减确认页。
3. 服务端根据记录快照生成建议扣减项。
4. 用户逐项确认或修改后提交。
5. 每条做饭记录最多成功应用一次扣减；重复请求返回同一结果。

扣减不能让已知数量变为负数。数量未知的库存默认不自动修改，用户可以手工填入剩余量。

## 4. 家庭管理

### 4.1 权限

- 创建家庭的人为 `OWNER`，另一名成员为 `MEMBER`。
- 双方可以修改家庭名称、查看成员和管理邀请码。
- `OWNER` 可以移除成员、转让管理权和解散家庭。
- `MEMBER` 可以主动退出。
- 两人家庭中的 `OWNER` 不能直接退出，必须先转让或解散。

### 4.2 退出和移除

- 退出者账号保留，回到创建或加入家庭状态。
- 家庭菜谱、库存和历史记录留在原家庭，退出者不再访问。
- 退出者对未完成菜单的选择与确认撤销，菜单版本递增并回到待确认状态。
- 已完成记录中的成员显示为“已退出成员”。
- 退出者已经发布的家庭菜谱留在家庭；个人草稿仍归退出者。
- 当前家庭所有未使用邀请码立即失效。

### 4.3 危险操作

- 退出和移除采用影响说明加二次确认。
- 转让管理权明确展示双方角色变化。
- 解散需要输入家庭名称，并重新通过微信身份校验。
- 解散删除家庭库存、共享菜谱、菜单和历史；双方账号保留。
- 所有危险操作使用幂等键、事务和一致的成员锁顺序。

## 5. 提示与消息

### 5.1 小程序内反馈

- 加载使用骨架或局部加载；保存期间禁用重复提交。
- 表单错误显示在对应字段附近，并保留用户输入。
- 区分断网、超时、服务繁忙、登录失效、权限不足和版本冲突。
- 同一业务错误码在所有页面使用统一文案和操作。
- 空状态说明原因并提供下一步，不只显示“暂无数据”。
- 危险操作必须展示影响的数据范围。

### 5.2 消息中心

“我的”页面增加消息入口，支持已读、全部已读和业务跳转。首版事件包括：

- 对方加入家庭。
- 对方更新今晚选择。
- 菜单等待确认或因修改需要重新确认。
- 菜单完成。
- 家庭菜谱或库存被修改。
- 管理权转让、成员退出或被移除。

消息正文不保存微信凭证、邀请码、访问令牌或不必要的菜谱自由文本。

### 5.3 微信订阅提醒

- 只在相关业务动作完成后，以场景化入口调用 `wx.requestSubscribeMessage`。
- 用户拒绝后仍保留小程序内消息，不反复打扰。
- 模板 ID 由管理员在微信公众平台配置，通过生产环境变量注入，禁止写死在仓库。
- 服务端记录发送幂等键、微信消息 ID 和结果，不把发送失败视为业务动作失败。
- 首版只做：等待选择或确认、重新确认、菜单完成、家庭角色变化。

## 6. 真实图片库

### 6.1 来源策略

- Pexels 为主，Pixabay 和明确公有领域的 Wikimedia Commons 图片补充。
- 不使用搜索引擎结果页、无来源转载图、带水印图片、品牌或人物可识别图片。
- 不接入面向用户的外部图库实时搜索，不直接使用第三方外链。
- 入库时保存原始页面、作者、许可、获取日期、文件哈希、宽高和内部对象路径。
- 即使许可不强制署名，仍在图片详情或关于页面提供来源信息。

参考许可：

- <https://www.pexels.com/license/>
- <https://pixabay.com/service/license-summary/>
- <https://commons.wikimedia.org/wiki/Commons:Reusing_content_outside_Wikimedia/en>

### 6.2 存储与交付

- 首版由自有服务托管优化后的 WebP/JPEG，不把图片打进小程序主包。
- 数据库存内部对象键，不存临时第三方 URL；API 生成 `https://www.osheeep.com` 下的稳定地址。
- 原图仅用于内部留档，客户端使用适配列表和详情尺寸的派生图。
- 图片服务接口保持存储实现无关，以后可以迁移到对象存储而不改菜谱数据。
- 正式提审前替换 8 道系统菜的现有生成图片，并完成逐图授权复核。

## 7. 后端数据设计

现有 `dinner_recipes`、菜单和记录表继续演进，不重建原有核心闭环。新增或扩展以下模型：

### 7.1 食材与库存

- `dinner_ingredients`：标准或家庭食材、名称、分类、默认单位、状态。
- `dinner_household_inventory`：家庭、食材、数量、单位、版本、最后修改者。
- 家庭和食材唯一约束防止重复库存项。

### 7.2 菜谱

- 扩展 `dinner_recipes`：`DRAFT/PUBLISHED/ARCHIVED`、份量、版本、来源菜谱、最后修改者、图片资源、发布时间和归档时间。
- `dinner_recipe_ingredients`：菜谱食材、数量、单位、是否必需和顺序。
- `dinner_recipe_methods`：做法名称、烹饪方式、预计时间、默认标记和状态。
- `dinner_recipe_method_steps`：有序步骤和说明。
- `dinner_image_assets`：来源、许可、作者、对象键、哈希、尺寸和审核状态。

### 7.3 历史和扣减

- 扩展做饭记录菜品快照，保存所选做法、食材用量和最终图片地址快照。
- `dinner_inventory_consumptions`：记录 ID、幂等键、应用状态和操作者。
- `dinner_inventory_consumption_items`：建议量、实际扣减量和扣减前后值。

### 7.4 家庭与消息

- 扩展家庭成员：角色、状态、加入/退出时间和版本。
- 扩展家庭：版本和管理员变更时间。
- `dinner_notifications`：接收者、类型、业务引用、已读时间和创建时间。
- `dinner_subscription_deliveries`：模板类型、幂等键、微信消息 ID、发送结果和时间。

所有数据库变化使用新的 Flyway 迁移；不得修改已经执行的 `V1` 至 `V4`。

## 8. API 边界

新增接口按资源拆分：

- `/api/dinner/ingredients`：标准食材搜索和家庭自定义食材。
- `/api/dinner/inventory`：家庭库存查询、批量保存和扣减预览。
- `/api/dinner/recipes`：找菜、详情、草稿、发布、复制和归档。
- `/api/dinner/recipes/{id}/methods`：做法版本管理。
- `/api/dinner/image-assets`：内部图片库查询。
- `/api/dinner/household/members`：成员、角色、退出、移除和转让。
- `/api/dinner/household/dissolution`：身份复核后的解散。
- `/api/dinner/notifications`：消息列表和已读。
- `/api/dinner/records/{id}/inventory-consumption`：扣减预览和幂等提交。

写接口统一使用版本或幂等键。服务端根据当前用户和家庭成员关系判断权限，不信任客户端传入的角色、家庭 ID 或创建者 ID。

## 9. 平台与合规边界

- 自定义菜名、做法和步骤是用户填写内容。发布到家庭前调用微信文本内容安全识别；管理员需要重新评估公众平台的用户生成内容声明。
- 官方文本检查能力用于识别违法违规文本：<https://developers.weixin.qq.com/miniprogram/dev/OpenApiDoc/sec-center/sec-check/msgSecCheck.html>。
- 微信订阅消息必须先在公众平台选择模板，再由用户通过弹窗授权并由服务端发送：<https://developers.weixin.qq.com/miniprogram/dev/framework/open-ability/subscribe-message.html>。
- 新增数据类型、订阅偏好和图片来源说明后，需要更新本地隐私政策及平台隐私保护指引；只有声明过的隐私接口才能调用：<https://developers.weixin.qq.com/miniprogram/dev/framework/user-privacy/PrivacyAuthorize.html>。
- 不开放用户上传图片，因此首版不新增相册权限、多媒体内容审核或用户文件删除流程。

## 10. 分阶段实施

### 阶段 A：菜谱与库存基础

- 数据迁移、食材字典、库存、菜谱食材和做法模型。
- 服务端权限、版本控制、匹配算法和内容安全适配层。
- 图片资源元数据和内部图库 API。

### 阶段 B：找菜与家庭菜谱体验

- 实现选定的找菜首页视觉方向。
- 食材库存页面、菜谱详情和分步编辑。
- 系统菜复制、草稿、发布、共同编辑和归档。
- 接入今晚菜单并保存做法与食材快照。

### 阶段 C：完成后的库存处理

- 扣减预览、修改、幂等提交和失败重试。
- 做饭记录展示最终食材和做法快照。

### 阶段 D：家庭管理

- 家庭资料、角色、退出、移除、转让和解散。
- 处理未完成菜单、邀请码、共享数据和历史身份显示。

### 阶段 E：提示、消息与订阅

- 统一反馈组件和错误文案。
- 消息中心。
- 平台模板配置完成后接入微信订阅提醒。

### 阶段 F：真实图片与正式版收口

- 建立首批真实图片库，替换全部系统菜图片。
- 隐私、内容安全和用户生成内容声明复核。
- 双账号、iPhone、Android、弱网、并发和数据迁移回归。
- 重新准备正式提审材料。

## 11. 验收标准

- 两名成员共享库存，数量和单位修改不会静默覆盖对方更新。
- 用户能够按库存匹配、临时食材和“只看能做”找到菜。
- 用户能够保存草稿、发布家庭菜谱、添加做法、复制系统菜和归档。
- 菜谱变更不影响已完成记录；完成记录保存最终做法与食材快照。
- 完成菜单后可以跳过或幂等扣减库存，任何扣减错误都不会产生重复记录。
- 成员退出、移除、转让和解散符合权限与数据归属规则。
- 所有共享自定义文本发布前经过内容安全检查。
- 正式产品中不存在生成式菜品图片，所有图片均有可追溯来源和许可记录。
- 核心成功、失败、空状态、冲突和危险操作都有明确反馈。
- 微信订阅提醒完全可选，拒绝不影响业务功能，也不会被反复索取。

## 12. 已识别但暂不纳入本轮的功能

以下功能与正式版有关，但不是当前核心链路的前置条件，实施前另行确认：

- 缺少食材一键加入购物清单。
- 家庭口味偏好、忌口和过敏原。
- 收藏、评分和最近常做。
- 一周菜单规划。
- 营养和热量计算。
- 菜谱公开分享或社区。
- 用户上传菜品图片。
- 食材批次、保质期和临期提醒。
- 三人及以上家庭和多人投票规则。

## 13. 非目标

- 不使用生成式图片替代真实菜品照片。
- 不在小程序内实时搜索或热链第三方图库。
- 不引入公开 UGC 社区。
- 不把库存扩展为采购、仓储和财务系统。
- 不改变固定 JAR、systemd 和浅层目录的生产部署方式。
- 未经管理员明确授权，不操作微信公众平台、生产服务器、提审或发布。
