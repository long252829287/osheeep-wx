# “今晚吃什么”隐私合规与账号注销设计

更新日期：2026-07-13

## 1. 目标

本阶段为微信正式提审补齐以下能力：

1. 登录前可访问的《用户协议》和《隐私政策》。
2. 明确、主动、未默认勾选的协议同意流程。
3. “我的”页面中的隐私与账户入口。
4. 使用当前微信身份重新验证的自助账号注销闭环。
5. 与真实代码和数据处理行为一致的微信公众平台配置清单。

本设计不直接提交微信审核，也不代替账号管理员在微信公众平台完成实名认证信息、正式名称、隐私保护指引、服务类目和审核材料配置。

## 2. 已确认的产品决定

- 采用小程序原生协议页面，不依赖 `web-view` 或额外业务域名。
- 用户主动勾选协议后，才能调用微信登录。
- 账号注销包含前端入口、微信身份复核、后端事务处理和旧令牌立即失效。
- 双人家庭中一方注销时，另一方的共同历史继续保留，但注销成员的身份必须匿名化。
- 最后一名成员注销后，删除该家庭及其关联的邀请码、菜单和做饭记录，不再保留无人使用的家庭业务数据。
- 微信公众平台《用户隐私保护指引》和隐私接口声明由账号管理员人工配置，内容必须与代码一致。

## 3. 当前事实基线

### 3.1 当前使用的微信能力

小程序当前使用：

- `wx.login`：获取临时登录凭证。
- `wx.request`：访问生产 API。
- `wx.getStorageSync`、`wx.setStorageSync`、`wx.removeStorageSync`：保存和清除访问令牌。
- `wx.getClipboardData`：仅在用户点击“粘贴邀请码”后读取剪贴板。
- `wx.setClipboardData`：仅在用户点击“复制邀请码”后写入剪贴板。
- 小程序分享能力：分享包含邀请码的加入路径。

当前不调用手机号、微信昵称头像、相册、摄像头、定位、通讯录或设备广告标识相关接口。

### 3.2 当前服务端数据

服务端当前保存：

- 微信 `openid` 与内部用户 ID 的绑定。
- 内部用户记录和账号状态。
- 家庭名称、成员关系和邀请码哈希。
- 菜品选择、菜单状态、确认与完成动作。
- 做饭记录和菜品快照。
- 必要的应用运行日志。

邀请码有效期为 24 小时，服务端只保存哈希。生产应用日志按当前配置最多保留 14 天。

### 3.3 当前缺口

- 登录页只有不可点击的协议说明文字。
- 登录前不能查看协议全文。
- 没有独立的隐私与账户页面。
- 没有账号注销接口和用户入口。
- JWT 只校验签名和有效期，不能在注销后立即识别账号已失效。
- 微信公众平台配置无法由代码变更替代，仍需管理员人工核对。

## 4. 前端信息架构

新增四个原生页面：

```text
pages/legal/user-agreement/index
pages/legal/privacy-policy/index
pages/privacy-center/index
pages/account-deletion/index
```

### 4.1 登录页

登录页调整为：

- 《用户协议》和《隐私政策》分别是可点击链接。
- 链接在未登录状态下可访问，返回后保留当前页面状态。
- 增加未默认选中的同意控件。
- 未同意时登录按钮不可用，并提供清晰的状态说明。
- 同意后点击登录，才调用 `wx.login` 和 `/api/auth/wechat`。
- 不因协议流程额外申请手机号、头像、相册或定位权限。

本阶段不新增服务端同意记录表。客户端以主动勾选作为本次登录动作的前置条件；协议版本变化时，应在后续版本中清除旧的本地同意状态并要求重新确认。

### 4.2 协议页面

《用户协议》至少包含：

- 服务内容和适用范围。
- 账号登录与使用规则。
- 双人家庭、邀请码和共享历史的规则。
- 用户内容和禁止行为。
- 服务变更、中断与责任边界。
- 账号注销和协议更新。
- 运营主体和联系方式。

《隐私政策》至少包含：

- 个人信息处理者名称和联系方式。
- 处理目的、方式、信息种类和保存期限。
- 微信登录、网络请求、本地令牌、家庭及菜单数据的说明。
- 剪贴板只在用户主动点击时读取的说明。
- 不收集的数据和未申请的权限。
- 共享、委托处理和安全措施说明。
- 查阅、更正、删除、撤回同意和注销的路径。
- 未成年人说明、政策更新和生效日期。

运营主体展示值记录为“个人主体姓名”。正式提审检查必须验证该展示值与微信公众平台实名认证姓名完全一致；不一致时禁止提交审核。

隐私联系邮箱为 `15203700590@163.com`。

### 4.3 隐私与账户页

“我的”页面新增“隐私与账户”入口。页面展示：

- 《用户协议》和《隐私政策》入口。
- 当前实际处理的信息类型和用途摘要。
- 运营主体与隐私联系邮箱。
- 查阅、更正、删除和投诉咨询方式。
- “注销账号”入口。

### 4.4 注销页

注销页必须先展示不可省略的影响说明：

- 原账号和原历史关联无法恢复。
- 当前微信身份绑定会被删除。
- 用户会退出当前家庭。
- 由该用户创建的有效邀请码会失效。
- 共同菜单和做饭历史会以“已注销成员”身份保留给另一方。
- 注销成功后本机登录状态会被清除。

用户完成二次确认后，客户端重新调用 `wx.login` 获取一次性临时凭证，再发起注销请求。注销成功后清除本地访问令牌并 `reLaunch` 到首次登录页。

## 5. 后端接口与身份复核

新增受 JWT 保护的接口：

```http
POST /api/users/me/deletion
Authorization: Bearer <token>
Content-Type: application/json

{
  "code": "fresh wx.login code"
}
```

选择动作式 `POST`，避免依赖带请求体的 `DELETE` 请求在各层的兼容性。接口返回项目统一的 `ApiResponse`。

处理顺序：

1. 解析 JWT 得到当前内部用户 ID。
2. 使用本次新取得的微信临时凭证调用 `code2session`。
3. 查询当前用户的微信身份绑定。
4. 比较 `openid`；不一致时拒绝注销，不执行任何数据变更。
5. 身份匹配后进入单个数据库事务完成注销。

微信临时凭证只能使用一次。前端请求失败后若需要重试，必须重新调用 `wx.login`，不能复用旧 `code`。

## 6. 注销事务与共享数据

注销事务执行：

1. 锁定当前用户、微信身份、家庭成员关系和相关家庭，避免并发加入、刷新邀请码或重复注销。
2. 删除 `wechat_user_identities` 中的身份绑定。
3. 删除当前用户的家庭成员关系。
4. 撤销该用户创建且仍有效的邀请码。
5. 将用户记录去标识化：
   - `email`、`password_hash`、`display_name`、`avatar_url` 置空。
   - `username` 替换为不包含原身份信息且保持唯一的系统值。
   - `status` 改为 `DELETED`。
   - 写入 `deleted_at`。
6. 保留满足另一名成员历史回看所必需的菜单、动作和做饭记录外键，但 API 不再把注销用户映射为可识别成员。
7. 若家庭仍有成员，保留家庭和共享历史，注销成员统一显示为“已注销成员”。
8. 若当前用户是家庭最后一名成员，按外键依赖顺序删除该家庭的菜品快照、做饭记录、菜单动作、菜品选择、菜单、邀请码和家庭记录。

任一步骤失败则整个事务回滚。事务完成后，同一 `openid` 再次登录会创建全新账号，不自动关联旧家庭或历史。

实现时不得把原 `openid`、用户名、家庭名称或访问令牌写入注销日志。

## 7. 旧令牌立即失效

当前 JWT 是无状态令牌，注销后仍可能在原有效期内通过签名校验。为确保立即失效，JWT 鉴权过滤器在解析令牌后必须验证数据库中的用户仍满足：

- 用户记录存在。
- `status` 为 `ACTIVE`。
- `deleted_at` 为空。

不满足时清除安全上下文并按未认证请求处理。该检查适用于所有受保护接口，因此被停用或注销的账号不能继续使用旧令牌。

本阶段接受每个受保护请求增加一次轻量用户状态查询。若未来流量需要优化，可在不改变鉴权语义的前提下增加短期缓存和主动失效机制。

## 8. 数据处理与留存规则

### 8.1 信息和用途

| 信息                 | 用途               | 保存原则                                     |
| -------------------- | ------------------ | -------------------------------------------- |
| 微信 `openid`        | 登录和区分账号     | 账号存续期间；注销时删除绑定                 |
| 本地访问令牌         | 维持登录状态       | 到期、未授权或注销时清除                     |
| 家庭名称和成员关系   | 建立双人家庭       | 仍有成员时保留；最后一名成员注销时删除       |
| 邀请码哈希           | 加入家庭           | 24 小时有效，过期或注销后不可使用            |
| 选菜、菜单和做饭记录 | 核心功能与历史回看 | 仍有成员时匿名化保留；最后一名成员注销时删除 |
| 剪贴板邀请码         | 帮助输入邀请码     | 仅用户点击时读取，不作为独立信息保存         |
| 应用运行日志         | 安全、排障和运维   | 当前最多 14 天，不记录密钥和完整身份标识     |

### 8.2 不收集范围

当前版本不收集手机号、真实姓名、微信昵称头像、相册内容、摄像头内容、精确位置、通讯录、广告标识或用户画像。

### 8.3 变更规则

以后新增个人信息种类、微信隐私接口、第三方 SDK 或处理目的前，必须先：

1. 更新代码级数据清单。
2. 更新《隐私政策》。
3. 更新微信公众平台《用户隐私保护指引》和隐私接口声明。
4. 判断是否需要重新取得同意。
5. 完成测试后再发布。

## 9. 微信公众平台人工清单

账号管理员在正式提审前完成并留存检查结果：

1. 正式小程序名称是否为“今晚吃什么”。
2. 个人主体实名认证姓名是否与协议中的运营主体完全一致。
3. AppID 是否为当前生产 AppID。
4. 体验版本是否为预期版本，首页是否为 `pages/onboarding/index`。
5. `request` 合法域名是否为 `https://www.osheeep.com`。
6. 《用户隐私保护指引》是否逐项声明实际处理的数据和用途。
7. 隐私接口声明是否只包含真实使用的接口；重点核对微信登录和用户主动触发的剪贴板读取。
8. 是否未声明手机号、头像昵称、相册、定位等未使用能力。
9. 服务类目、主体信息、开发者和体验成员是否正确。
10. 审核说明是否提供登录、创建/加入家庭、选菜、完成记录、查看隐私政策和注销账号的完整路径。

涉及管理员扫码、二次验证、名称申请和最终“提交审核”的操作由用户完成，自动化不得绕过平台安全限制。

## 10. 错误处理与安全要求

- 未勾选协议时不调用 `wx.login`。
- 协议页面加载不依赖登录和业务 API。
- 注销请求缺少或复用无效微信凭证时，提示用户重新验证。
- `openid` 与当前账号不一致时返回稳定业务错误码，不暴露实际标识。
- 已注销账号使用旧 JWT 时返回未认证结果。
- 重复注销不恢复、不重复创建或篡改历史数据。
- 注销事务并发冲突时回滚并返回可重试提示。
- 客户端只有在服务端确认注销成功后才清除本地状态。
- 日志不得记录微信临时凭证、`openid`、JWT、邀请码明文或其他密钥。

## 11. 测试设计

### 11.1 小程序

- 未同意协议时登录按钮不可用且不调用登录端口。
- 协议链接在登录前可访问并能返回。
- 同意后保持现有登录和路由行为。
- 剪贴板只在点击“粘贴邀请码”后读取。
- 注销影响说明和二次确认完整显示。
- 注销时获取新的微信临时凭证。
- 注销成功后清除访问令牌并返回登录页。
- 注销失败时保留登录状态并显示可理解错误。

### 11.2 后端

- 正确 JWT 与匹配 `openid` 可以注销。
- 微信身份不匹配时不修改任何数据。
- 无效或已使用的微信临时凭证被拒绝。
- 单成员家庭注销后，家庭及其关联业务数据被删除。
- 双成员家庭一方注销后另一方仍能查看匿名共享历史。
- 有效邀请码在创建者注销后失效。
- 用户字段完成去标识化，身份绑定被删除。
- 注销前签发的 JWT 在事务完成后立即失效。
- 事务中任一写操作失败时全部回滚。
- 同一微信再次登录时创建全新账号且不恢复旧历史关联。

## 12. 验收标准

- 登录前可以完整阅读用户协议和隐私政策。
- 未主动同意协议不能登录。
- 页面文案、微信平台声明与真实代码处理行为一致。
- 用户能在“我的 → 隐私与账户”找到联系和注销入口。
- 注销前必须通过当前微信身份重新验证。
- 注销后旧令牌立即失效，个人身份字段被清除。
- 共同历史对另一成员可用，但注销用户只显示为“已注销成员”。
- 自动化测试、TypeScript、ESLint、Prettier 和后端测试全部通过。
- 微信公众平台人工清单完成前，不点击“提交审核”。

## 13. 非目标

本阶段不包含：

- 获取手机号、头像昵称、相册或定位。
- 自定义菜谱、图片上传或家庭管理扩展。
- 公网 `web-view` 协议站点。
- 订阅消息、广告、用户画像或第三方统计 SDK。
- 自动操作微信公众平台的扫码、二次验证、名称申请或提交审核。
