# 家庭菜谱接入找菜与今晚菜单设计

更新日期：2026-07-21（Asia/Shanghai）

状态：产品范围、整体结构、数据权限和页面验收已由用户逐段确认。

## 1. 背景与目标

家庭自定义菜谱首个竖切已经支持个人草稿、分步自动保存、食材、默认做法、已审核图片、预览和内容安全发布恢复，但当前找菜接口只返回系统菜，今晚菜单也只接受 `SYSTEM + PUBLISHED` 菜谱。

本任务交付一个完整但克制的端到端竖切：

1. 当前家庭已发布的家庭菜谱与系统菜一起参与库存匹配和找菜排序。
2. 用户可以直接把家庭菜谱加入今晚菜单。
3. 菜单选择保存菜谱版本和默认做法身份。
4. 完成菜单时保存菜名、图片、食材、默认做法、步骤和菜谱版本快照。
5. 后续菜谱变化不得改写已经完成的做饭记录。

本任务不重新设计找菜页，也不扩大为完整菜谱管理或多做法冲突任务。

## 2. 已确认的产品决策

- 家庭菜谱和系统菜使用同一个发现列表、同一套库存匹配规则和同一套排序规则。
- 家庭菜不单独置顶，也不增加筛选器；仅显示“自家菜谱”文字标签。
- 加入菜单时只使用家庭菜谱当前的默认做法，不在本任务提供做法选择。
- 客户端继续提交当前用户完整的 `recipeIds` 集合；默认做法和菜谱版本由服务端解析并持久化。
- 完成菜单时写入完整历史快照；记录详情只读快照，不回查当前菜谱聚合。
- 系统菜没有默认做法数据时保持现有体验，不伪造做法名称或步骤。
- 数据库演进从 V7 开始，不修改 V1–V6。

## 3. 方案比较与选择

### 3.1 采用：扩展现有统一发现接口

扩展 `GET /api/dinner/recipes`，由服务端一次性加载系统菜和当前家庭已发布菜谱，再统一计算匹配、过滤和排序。

优点：

- 权限、筛选、排序和图片解析只有一套真相来源。
- 小程序不需要协调两个请求或自行合并不同响应结构。
- 现有找菜页加载、重试、只看能做和加入菜单交互可以原位演进。

### 3.2 不采用：客户端合并两个列表接口

分别请求系统菜发现接口与家庭菜谱列表，再由客户端补匹配和排序。这会复制服务端规则，使权限、图片、加载失败和排序更容易不一致。

### 3.3 不采用：新建独立统一目录接口

新建 `/api/dinner/catalog` 可以获得更干净的新契约，但会与现有发现接口重复，并迫使客户端和测试整体迁移，超出当前竖切所需范围。

## 4. 范围

### 4.1 本任务包含

- 统一发现系统菜和当前家庭已发布菜谱。
- 家庭菜谱库存匹配、临时包含/排除食材和“只看能做”。
- “自家菜谱”标签及其无障碍文案。
- 家庭菜谱加入今晚菜单、双方选择合并和菜单版本冲突恢复。
- 菜单选择中的菜谱版本与默认做法身份。
- 今晚菜单中的家庭菜来源和默认做法摘要。
- 完成时的食材、做法、步骤、版本和基本信息快照。
- 做饭记录详情展示新增快照内容。
- V7 迁移、服务端/小程序自动化、显式 MySQL 集成测试和原生模拟器 QA。

### 4.2 本任务不包含

- 已发布菜谱共同编辑。
- 其他做法变体、加入菜单时选择做法或双方做法冲突决策。
- 系统菜复制、家庭菜谱归档或恢复归档。
- 菜谱详情新页面。
- 双方偏好参与发现排序。
- 库存删除、单位选择、自定义食材或完成后的库存扣减。
- 替换其余 8 张系统菜开发期占位图。
- 真实微信内容安全成功联调、第二账号补证、生产部署、体验版上传、提审或发布。

## 5. 服务端架构

### 5.1 统一目录查询

`DinnerRecipeService.discover` 继续作为找菜目录的唯一入口。它根据当前用户的活跃家庭身份查询：

- `scope = SYSTEM AND status = PUBLISHED`；或
- `scope = HOUSEHOLD AND household_id = 当前家庭 AND status = PUBLISHED`。

服务端批量加载菜谱食材、家庭库存、家庭菜谱默认做法和已审核图片，避免按菜谱逐条查询。系统菜继续使用 `image_path`；家庭菜通过 `image_asset_id` 解析为内部已审核列表图 URL。客户端只收到可直接展示的 `imagePath`，不接触原始第三方文件地址或内部对象键。进入 `APPROVED` 状态的派生图对象键与内容不可原位覆盖；换图必须创建新资产，保证历史中保存的 URL 仍指向同一内容。

统一目录沿用现有排序：匹配状态、匹配比例、预计耗时、稳定菜谱 ID。本任务不伪造尚不存在的双方偏好排序信号。

### 5.2 菜谱目录组装边界

新增一个聚焦的目录组装边界，负责把菜谱实体、食材行、默认做法和图片资产组合为 `RecipeResponse`。`DinnerRecipeService` 负责身份、查询、库存匹配、过滤和排序；组装边界负责响应字段，不把家庭菜谱聚合装载继续堆进已有服务方法。

发布完成的家庭菜谱按现有发布校验必然具备名称、分类、口味、预计耗时、至少一个食材、默认做法、至少一个步骤和已审核图片。如果发现损坏的已发布聚合，服务端不得把草稿字段或跨家庭数据降级返回；该菜谱不进入可选择结果，并记录不含用户输入原文的结构化错误。

### 5.3 菜单选择

`PUT /api/dinner/menus/today/selections` 保持现有请求体：

```json
{
  "recipeIds": [1, 14],
  "version": 3
}
```

在持有今日菜单行锁并通过菜单版本校验后，服务端批量验证完整菜谱集合：

- 系统菜必须为 `SYSTEM + PUBLISHED`。
- 家庭菜必须为 `HOUSEHOLD + PUBLISHED`，并属于今日菜单的家庭。
- 草稿、归档菜谱、其他家庭菜和不存在的 ID 统一返回 `DINNER_RECIPE_INVALID`，不得泄漏其真实状态或归属。

服务端为每个家庭菜解析唯一的活动默认做法，验证该方法的 `recipe_id` 与菜谱一致，并在重建当前用户选择行时保存 `recipe_version` 和 `method_id`。系统菜当前没有做法聚合时保存 `method_id = NULL`，`recipe_version = 1`，继续兼容原闭环。

### 5.4 今晚菜单读取

`TodayMenuResponse.dishes` 增加菜谱来源、选择时菜谱版本和已保存默认做法摘要。菜单读取基于已经保存的选择身份，不允许客户端传入家庭 ID、菜谱来源或做法身份。本任务没有已发布菜谱编辑或其他做法入口，因此同一菜单、同一菜谱的所有选择行必须具有相同 `recipe_version` 和 `method_id`；发现不一致时按聚合损坏失败，不能任意选择其中一行展示。

本任务尚未开放编辑已发布菜谱和归档操作，因此活动菜单冻结完整内容的规则留给后续编辑/归档规格；V7 先保存版本与方法身份，为后续修订模型提供稳定引用。已完成记录的不可变性由完成时快照完整保证。

### 5.5 完成与历史快照

菜单完成仍使用现有菜单行锁、菜单版本和 UUID v4 幂等键。服务端在创建做饭记录前批量装载所有选择对应的菜谱、食材、已保存默认做法、步骤和已审核图片资产。家庭菜方法必须使用选择行中保存的 `method_id`，不能在完成时重新猜测“当前默认方法”；家庭菜图片通过 `image_asset_id` 批量解析，并把不可变列表图 URL 写入快照表已有的 `image_path` 列，不新增重复图片列。

完成时必须再次校验聚合关联，而不能只信任选择写入时的检查：家庭菜的 `household_id` 必须等于菜单家庭，方法的 `recipe_id` 必须等于选择的菜谱 ID，同一菜谱的选择行必须具有相同版本和方法，且当前菜谱版本必须等于选择行保存的 `recipe_version`。本任务没有编辑或归档已发布菜谱的入口，因此任何不一致均视为聚合损坏并在写记录前失败；后续编辑/归档规格必须重新定义活动菜单的版本冻结规则。

同一事务内按以下顺序执行：

1. 检查已有做饭记录并返回幂等结果。
2. 校验菜单版本与 `CONFIRMED` 状态。
3. 完整装载所有菜谱聚合；任何缺失均在写记录前失败。
4. 创建做饭记录。
5. 为每道菜写入完整快照。
6. 标记菜单完成并记录完成动作。

事务中任一快照写入失败必须整体回滚，不得留下半条记录、部分菜品或已完成但无快照的菜单。记录列表和详情只读取 `dinner_record_dish_snapshots`，不回查当前菜谱、方法或图片资产。

## 6. V7 数据设计

新增迁移 `V7__connect_household_recipes_to_menus.sql`，只做向前兼容的增量修改。

### 6.1 `dinner_menu_selections`

新增：

- `recipe_version BIGINT NOT NULL DEFAULT 1`
- `method_id BIGINT NULL`
- `KEY idx_dinner_selections_method (method_id)`
- `FOREIGN KEY (method_id) REFERENCES dinner_recipe_methods(id)`

`method_id` 允许为空以兼容当前系统菜和已有选择。家庭菜新选择必须保存活动默认做法 ID；业务服务不能依靠数据库可空性绕过该规则。

### 6.2 `dinner_record_dish_snapshots`

新增：

- `recipe_scope VARCHAR(16) NULL`
- `recipe_version BIGINT NULL`
- `servings INT NULL`
- `method_id BIGINT NULL`
- `method_name VARCHAR(64) NULL`
- `cooking_style VARCHAR(32) NULL`
- `method_steps JSON NULL`
- `ingredients JSON NULL`

历史快照不对 `method_id` 建外键；历史必须在方法或菜谱未来变化后独立存在。旧记录保留空值，API 将空 JSON 快照归一化为空数组。

图片继续复用现有 `image_path` 快照列：系统菜写入原系统图片路径，家庭菜写入完成时解析出的内部不可变列表图 URL。历史读取不得根据 `recipe_id` 或 `image_asset_id` 重新解析图片。

新记录的 `method_steps` 结构固定为：

```json
[
  {
    "instruction": "锅中炒匀",
    "sortOrder": 0
  }
]
```

新记录的 `ingredients` 结构固定为：

```json
[
  {
    "ingredientId": 2,
    "name": "鸡蛋",
    "quantity": 2,
    "unit": "枚",
    "required": true,
    "sortOrder": 0
  }
]
```

JSON 数组按 `sortOrder` 升序持久化。数量为空继续表示“适量”，不得被序列化为零。

### 6.3 迁移兼容

- V1–V6 保持不可变。
- 已有系统菜和旧菜单选择使用默认 `recipe_version = 1`。
- 旧做饭记录新增字段为空时仍按旧响应正常展示。
- V7 不删除或重命名现有列，不要求一次性重写旧历史 JSON。
- 本任务不执行生产 Flyway。正式发布前仍须解决 V6 后旧 JAR 查询 `ACTIVE` 的不兼容问题；不得把“只回滚旧 JAR”当作可用回滚方案。

## 7. API 契约

### 7.1 发现响应

`RecipeResponse` 保留现有字段并增加：

```json
{
  "scope": "HOUSEHOLD",
  "version": 8,
  "defaultMethod": {
    "id": 21,
    "name": "家常做法",
    "cookingStyle": "炒"
  }
}
```

新增独立的 `RecipeMethodSummaryResponse(id, name, cookingStyle)`，供发现和今晚菜单响应复用。不得复用现有包含 `steps` 的 `RecipeMethodResponse`，避免无意把详情步骤带入列表契约。

系统菜允许 `defaultMethod = null`。发现响应不返回步骤，避免找菜页携带详情级负载。

### 7.2 今晚菜单响应

`MenuDishResponse` 增加：

- `scope`
- `recipeVersion`
- `method`：`RecipeMethodSummaryResponse`，可空

`selectedRecipeIds` 和现有计数、来源、确认/完成字段保持兼容。

### 7.3 记录详情响应

`RecordDishResponse` 增加：

- `scope`
- `recipeVersion`
- `servings`
- `method`：`id`、`name`、`cookingStyle`、有序 `steps`
- 有序 `ingredients`

记录详情使用独立的 `RecordMethodSnapshotResponse` 和 `RecordIngredientSnapshotResponse`，不复用菜谱编辑器 DTO。旧记录产生于家庭菜接入菜单之前，服务端固定归一化为：`scope = SYSTEM`、`recipeVersion = 1`、`servings = null`、`method = null`、`ingredients = []`。客户端不得因为旧记录没有新增快照字段而显示错误页。

## 8. 小程序体验

### 8.1 找菜页

- 保持当前暖白背景、橄榄绿信息色、橙色加入动作和现有卡片层级。
- 系统菜与家庭菜统一排序，不新增分区、筛选器或置顶逻辑。
- 家庭菜在菜名附近显示“自家菜谱”文字标签；标签不能只依赖颜色表达。
- 主推荐和轻量菜谱行沿用现有加入、已加入、保存中和重试状态。
- 家庭菜的无障碍名称包含“自家菜谱”，触控目标继续至少 `88rpx`。

### 8.2 今晚菜单

- 家庭菜显示“自家菜谱 · 默认做法名”。
- 系统菜没有方法时保持现有元信息，不显示空占位或“未知做法”。
- 双方选择同一家庭菜继续合并为一行，来源为 `BOTH`。
- 菜单版本、确认、修改后重新确认和完成交互保持现有语义。

### 8.3 做饭记录详情

- 家庭菜显示“自家菜谱”标签。
- 有方法快照时展示方法名、烹饪方式和有序步骤。
- 有食材快照时展示名称、数量、单位和“适量”语义。
- 旧记录没有新字段时保持现有基本菜品卡，不出现空方法区或空食材区。

本任务基于已有找菜、今晚菜单和记录详情页面演进，不新增路由，也不重新选择视觉方向。

## 9. 错误与恢复

- `DINNER_MENU_VERSION_CONFLICT`：刷新最新菜单，保留 `pendingRecipeId`，显示冲突文案并要求用户明确重试；不得自动重放写请求。
- `DINNER_RECIPE_INVALID`：显示“这道家庭菜谱已不可用，请刷新后重试”，刷新发现结果与菜单，不泄漏菜谱状态或家庭归属。
- 目录整体加载失败：沿用当前错误页和“重新加载”。
- 已有结果刷新失败：保留旧结果并显示局部重试，不清空已加入状态。
- 完成快照失败：完成事务回滚，客户端保持可重试状态；相同幂等键不得生成重复记录。
- 图片 URL 缺失：不绕过发布完整性或跨家庭权限；服务端按损坏聚合处理，不把第三方原始 URL 暴露给客户端。

## 10. 安全与不变式

- 家庭身份、家庭 ID、菜谱来源、菜谱版本和默认做法均由服务端根据当前登录用户解析。
- 只有当前家庭的已发布家庭菜能够进入新的菜单选择。
- 草稿只有创建者可见，但永远不进入发现或菜单接口。
- 菜单写操作必须校验菜单版本；冲突后不自动重放。
- 完成动作继续使用 UUID v4 幂等键和“一菜单一记录”唯一约束。
- 历史详情完全来自完成时快照；后续菜谱、做法、食材、图片或成员变化不能改写历史。
- 数量为空表示“适量”，不是零、删除或缺失数据。
- 菜谱和方法的聚合装载必须批量执行，避免列表和完成路径出现 N+1 查询。

## 11. 测试与验收

### 11.1 服务端自动化

- V7 持久化契约：选择版本/方法字段、历史方法步骤与食材 JSON、旧数据兼容。
- 统一发现：系统菜与当前家庭菜共同返回，其他家庭、草稿和归档不返回。
- 统一匹配：临时包含/排除、“只看能做”和稳定排序同时适用于家庭菜。
- 图片解析：家庭菜只返回已审核内部派生图 URL。
- 菜单保存：家庭菜版本与默认方法落库；系统菜空方法兼容。
- 权限：跨家庭、草稿、归档和不存在菜谱统一拒绝。
- 菜单响应：双方选择同菜合并，来源与方法摘要正确。
- 完成快照：食材、步骤、方法、版本、图片和选择人全部落库。
- 事务与幂等：快照失败整体回滚；重复完成只返回同一记录。
- 旧记录详情：新增字段为空时返回兼容响应。

### 11.2 显式 MySQL 8 集成

显式 MySQL 测试必须覆盖三条迁移路径：

1. 空库执行 V1–V7，证明全新安装可建立完整结构。
2. 使用 V4 结构及旧菜单、记录、系统菜夹具构造的生产形态测试库连续执行 V5、V6、V7，证明旧数据可升级；不得复制真实生产数据或连接生产库。
3. 当前专用 V6 测试库执行 V7，证明本地竖切基线可增量升级。

三条路径随后共同验证：

1. 两个家庭用户都能发现同一条本家庭已发布菜谱。
2. 其他家庭用户不能发现或加入该菜谱。
3. 双方选择同一家庭菜后菜单合并为一行。
4. 完成前篡改家庭、方法归属或选择版本会整笔拒绝，不写做饭记录。
5. 完成后快照包含确定顺序的食材、默认做法、步骤和内部图片 URL。
6. 完成后修改菜谱、方法、步骤、食材和图片关联夹具，不会改变已写入的历史快照响应。

测试必须继续通过进程环境、实际 JDBC catalog 和 Flyway data source/schema 安全门；不得指向共享或生产数据库。

## 12. 完成定义

只有同时满足以下条件，本地任务才可标记完成：

1. 当前家庭已发布菜谱能够出现在统一找菜结果中并正确匹配库存。
2. 家庭菜可以加入今晚菜单，双方选择能够合并，跨家庭或非发布菜不能加入。
3. 菜单选择保存菜谱版本与默认做法身份。
4. 完成菜单后记录详情展示不可变的食材、默认做法和步骤快照。
5. 现有系统菜、菜单并发、确认、完成和旧记录全部兼容。
6. 后端、小程序、显式 MySQL 和原生三视口 QA 全部达到本设计的验收门槛。
7. 文档明确记录：本地通过不等于真实微信内容安全、生产迁移、体验版或正式发布通过。

## 13. 后续任务接口

后续“已发布菜谱共同编辑、其他做法和归档”规格可以在本任务保存的 `recipe_version` 与 `method_id` 上继续演进，但必须另行定义：

- 活动菜单在菜谱修订或归档后的冻结规则。
- 双方选择不同做法时的冲突与最终做法决策。
- 发布修订与历史版本的关系。
- 归档后已有未完成菜单的展示和修改规则。

这些规则不得在当前竖切中以临时分支或客户端猜测实现。
