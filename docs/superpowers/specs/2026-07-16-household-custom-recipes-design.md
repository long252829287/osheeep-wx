# 家庭自定义菜谱设计

更新日期：2026-07-16

## 1. 目标与范围

本设计为“小家开饭”增加家庭自定义菜谱。用户可以创建仅自己可见的草稿，分步填写基本信息、食材、默认做法和真实图片，经过服务端内容安全检查后发布为家庭共享菜谱。发布后家庭双方均可查看和编辑；后续增量支持做法变体、系统菜复制、归档、菜谱详情、做法选择和今晚菜单联动。

本设计定义完整的自定义菜谱领域边界，但第一个实施计划只交付以下端到端竖切：

> 个人草稿 → 自动保存 → 选择食材 → 默认做法 → 选择已审核真实图片 → 预览发布 → 家庭菜谱列表双方可见

首个竖切不接入找菜排序、今晚菜单、做法冲突、系统菜复制、归档操作或已发布菜谱的共同编辑界面。这些能力必须沿用本文的数据模型、权限和版本边界增量实现，不得另建第二套菜谱模型。

## 2. 已确认的产品规则

- 产品继续限定两人家庭，不扩展为公开菜谱社区。
- 新建菜谱先成为个人草稿，只有创建者可见。
- 草稿可以没有图片；发布必须绑定一张状态为 `APPROVED` 的真实图片资产。
- 家庭成员不能上传图片，也不能审核图片；只能从内部真实图库选择。
- 发布后菜谱归当前家庭共享，双方都可查看和编辑。
- 编辑已发布菜谱时先形成个人编辑稿；内容安全检查和版本检查通过后，才原子替换共享版本。
- 系统菜只读，可以复制为独立的家庭草稿，并保留来源关联。
- 删除采用归档；归档不改写已有菜单和做饭记录快照。
- 每道已发布菜谱必须且只能有一个默认做法，可以增加其他做法变体。
- 菜谱食材数量可以为空，表示“适量”；单位与必需/可选属性仍然必填。
- 菜名、做法、步骤和其他共享自由文本在发布或更新前由服务端调用微信内容安全能力自动检查。
- 家庭内部不增加人工审批。任一有权限的家庭成员都可以提交发布或更新；检查通过后直接生效。
- 内容检查失败或暂时不可用时保留草稿或编辑稿，不修改共享版本。

## 3. 领域模型

### 3.1 菜谱聚合

继续演进现有 `dinner_recipes`，不新增平行的菜谱主表。菜谱是聚合根，基本信息、食材、做法、步骤和图片的写入统一受菜谱 `version` 保护。

V6 将现有系统菜从 `ACTIVE` 迁移为 `PUBLISHED`，家庭菜使用以下状态：

- `DRAFT`：个人新建草稿或个人编辑稿。
- `PUBLISHED`：家庭共享或系统只读菜谱。
- `ARCHIVED`：已归档家庭菜谱，不参与找菜和今晚菜单候选。

`dinner_recipes` 增加或明确以下字段：

| 字段 | 语义 |
| --- | --- |
| `scope` | `SYSTEM` 或 `HOUSEHOLD` |
| `status` | `DRAFT`、`PUBLISHED` 或 `ARCHIVED` |
| `household_id` | 当前发布目标或共享菜谱所属家庭；个人离家后的草稿可以为空 |
| `creator_id` | 草稿及菜谱的原始创建者 |
| `last_modified_by` | 最近一次成功保存或发布的操作者 |
| `servings` | 份量，发布时必须为正整数 |
| `version` | 聚合乐观版本，新建持久化记录从 `1` 开始 |
| `image_asset_id` | 关联内部图片资产；草稿可空，发布时必填且必须为 `APPROVED` |
| `source_recipe_id` | 从系统菜复制时记录源菜谱；复制后独立演进 |
| `revision_of_recipe_id` | 编辑稿对应的已发布家庭菜谱 |
| `base_published_version` | 创建编辑稿时共享菜谱的基线版本 |
| `published_at` | 首次成功发布的时间 |
| `archived_at` | 成功归档的时间 |

同一用户对同一已发布菜谱最多保留一个未发布编辑稿。新建草稿没有 `revision_of_recipe_id`。编辑稿的内容对另一名家庭成员不可见，直到更新发布成功。

### 3.2 菜谱食材

沿用 `dinner_recipe_ingredients`，每行保存食材、数量、单位、必需或可选、排序位置。一个菜谱不能重复添加同一食材。

- `quantity` 允许为空，客户端显示“适量”。
- `unit` 不允许为空，即使数量是“适量”也必须保留展示单位。
- 数量为空时，库存匹配只判断该食材是否存在，不比较库存数量。
- 食材步骤保存采用整组替换语义，在同一事务内校验、删除旧行并按顺序写入新行，同时递增菜谱版本。

### 3.3 做法与步骤

新增 `dinner_recipe_methods` 和 `dinner_recipe_method_steps`。

`dinner_recipe_methods` 保存做法名称、烹饪方式、预计耗时、是否默认、状态和排序。`dinner_recipe_method_steps` 保存所属做法、有序步骤和步骤说明。

- 已发布菜谱必须且只能有一个默认做法。
- 默认做法至少包含一个非空步骤。
- 首个竖切只创建默认做法；其他做法变体在后续竖切增加。
- 菜谱列表使用默认做法的预计耗时作为主要耗时；为了保持现有发现接口兼容，发布事务同步更新 `dinner_recipes.estimated_minutes`。
- 删除或重排步骤属于菜谱聚合写入，必须携带菜谱期望版本。

### 3.4 图片资产

新增 `dinner_image_assets`，至少保存：

- 来源平台和原始页面 URL；
- 作者或上传者；
- 许可名称和许可页面 URL；
- 获取日期；
- 原始文件 SHA-256；
- 原图宽高；
- 内部原图、列表派生图和详情派生图对象键；
- 审核状态与审核时间。

审核状态为 `PENDING / APPROVED / REJECTED / DISABLED`。小程序查询接口只返回 `APPROVED` 资产。客户端使用自有域名下的稳定派生图，不保存或热链第三方临时 URL。

家庭成员没有图片上传、图片审核或修改许可元数据的接口。图片入库属于内部受控流程，首个竖切至少需要一张来源、作者、许可、哈希和尺寸完整的真实照片用于端到端验收；测试环境可以使用明确标记的测试资产记录，但正式候选不能把测试记录或生成图当作真实照片。

## 4. 生命周期与权限

### 4.1 新建草稿

当前有活跃家庭成员关系的用户可以创建草稿。服务端从登录身份推导用户和当前家庭，不接受客户端指定创建者或任意家庭 ID。

新建草稿只有创建者可以查询和修改。草稿保存不调用外部内容安全服务。创建者未来退出家庭时，家庭管理事务将草稿与原家庭解绑但保留创建者所有权；创建者加入新家庭后可以重新绑定发布目标。已发布菜谱不会随创建者离开而迁移。

### 4.2 首次发布

首次发布必须同时满足：

- 操作者是草稿创建者，并且仍是目标家庭的活跃成员；
- 基本信息完整，名称、分类、口味、份量和预计耗时合法；
- 至少有一个必需食材，所有食材的单位和必需/可选属性合法；
- 存在且只存在一个默认做法，默认做法至少有一个有效步骤；
- 已绑定 `APPROVED` 图片资产；
- 所有需要共享的自由文本通过内容安全检查；
- 请求的 `expectedVersion` 等于数据库当前版本。

发布在一个事务中完成状态转换、家庭归属确认、发布时间、最后修改者和版本递增。任何校验或外部检查失败都不改变草稿状态与内容。

### 4.3 编辑已发布菜谱

当前家庭双方都可以基于已发布菜谱创建自己的个人编辑稿。编辑稿复制共享菜谱的完整聚合，记录 `revision_of_recipe_id` 和 `base_published_version`，后续自动保存只影响编辑稿。

更新发布时，服务端先执行完整性、权限、图片和文本安全检查，再锁定共享菜谱并比较 `base_published_version`。如果共享版本已变化，返回版本冲突并保留编辑稿；客户端必须展示新版本并让用户显式决定如何调整，不能自动重放或覆盖。

检查通过时，在一个事务中用编辑稿内容替换共享聚合、递增共享版本、记录最后修改者，并删除或标记该编辑稿已应用。

### 4.4 系统菜复制与归档

系统菜始终只读。复制操作创建当前用户可见的家庭草稿，复制基本信息、食材、默认做法、其他做法和当前图片关联，并通过 `source_recipe_id` 保留来源。

已发布家庭菜谱的归档属于家庭日常共享操作，任一活跃家庭成员都可以执行。归档必须携带期望版本并二次确认影响：菜谱将退出找菜和今晚菜单候选，但已有菜单引用和做饭记录快照不变。首个正式版本不提供恢复归档，避免引入未确认的恢复语义。

## 5. API 边界

保留现有 `GET /api/dinner/recipes` 找菜接口的默认行为，避免首个竖切破坏库存匹配和现有小程序。家庭菜谱写入与列表使用明确的子资源：

| 方法与路径 | 用途 |
| --- | --- |
| `GET /api/dinner/recipes/family?tab=PUBLISHED|DRAFT|ARCHIVED` | 查询家庭共享菜谱、我的草稿或已归档菜谱 |
| `POST /api/dinner/recipes/drafts` | 创建个人草稿 |
| `GET /api/dinner/recipes/{id}` | 按权限读取菜谱、草稿或编辑稿详情 |
| `PUT /api/dinner/recipes/{id}/basic-info` | 整体保存基本信息与期望版本 |
| `PUT /api/dinner/recipes/{id}/ingredients` | 整体替换有序食材与期望版本 |
| `PUT /api/dinner/recipes/{id}/default-method` | 整体保存默认做法、步骤与期望版本 |
| `PUT /api/dinner/recipes/{id}/image` | 选择已审核图片与期望版本 |
| `POST /api/dinner/recipes/{id}/publish` | 首次发布草稿或应用编辑稿 |
| `POST /api/dinner/recipes/{id}/edit-drafts` | 为已发布家庭菜谱创建或返回当前用户编辑稿 |
| `POST /api/dinner/recipes/{id}/copy` | 将系统菜复制为个人草稿 |
| `POST /api/dinner/recipes/{id}/archive` | 归档家庭菜谱 |
| `GET /api/dinner/image-assets` | 搜索可供选择的 `APPROVED` 图片资产 |

所有写接口由服务端根据当前身份计算用户、家庭和权限。客户端不能提交可信角色、创建者或家庭 ID。写响应返回最新完整版本号；前端只能用服务端响应推进本地版本。

## 6. 自动保存与页面数据流

首个竖切新增三个微信原生小程序页面：

1. 家庭菜谱列表：`已发布 / 我的草稿 / 已归档` 三个标签页。
2. 分步菜谱编辑器：基本信息、食材、默认做法、真实图片、预览发布五步。
3. 真实图片选择：搜索内部图库并查看来源信息。

编辑器在有效输入停止约 800ms 后自动保存当前步骤。点击下一步、返回或页面隐藏前强制完成当前保存。实现必须满足：

- 同一草稿的写请求串行发送；
- 保存期间产生的新修改合并到下一次保存；
- 不取消已经到达服务端的写请求；
- 不并发使用同一个期望版本；
- 409 冲突后停止自动保存，保留本地输入并要求用户刷新或重新调整；
- 普通网络错误保留输入并提供明确重试；
- 发布请求绝不自动重放。

列表中的已发布菜谱显示创建者、最后修改者和更新时间；草稿显示当前完成步骤；归档项弱化展示。预览步骤使用最终家庭成员可见的层级，并逐项列出尚不能发布的原因。

## 7. 视觉与交互原则

- 延续已批准的库存和找菜方向：暖白背景、橄榄绿层级、橙色主动作和微信原生导航。
- 不重新选择整体视觉方向，也不把每个字段包成悬浮卡片。
- 食材使用现有目录搜索添加；数量为空显示“适量”。
- 默认做法支持步骤新增、删除和排序。
- 保存状态明确显示“保存中、已保存、保存失败、版本冲突”，不能只靠颜色区分。
- 字段错误贴近字段展示，切换步骤和请求失败均保留输入。
- 主要触控目标至少 `88rpx`。
- 实现前基于现有正式版视觉源制作自定义菜谱参考图；实现后在微信开发者工具用 375、390、430px 对同一状态做参考图与实现图比较。
- 菜品成品照只能使用授权来源可追溯的真实照片，禁止生成图、无来源转载、搜索结果页图片和第三方热链。

## 8. 内容安全

后端定义存储无关的内容安全网关。生产实现调用微信文本内容安全能力；测试注入确定性实现。测试或本地开发实现不得在生产配置中自动启用。

检查内容包括菜名、口味自由文本、做法名称、烹饪方式、步骤和备注。服务端对文本进行稳定归一化并一次提交发布所需内容，不在草稿每次输入时调用外部检查。

- 检查不通过：HTTP 422，业务码 `DINNER_RECIPE_CONTENT_REJECTED`。
- 服务暂不可用或生产未配置：HTTP 503，业务码 `DINNER_RECIPE_MODERATION_UNAVAILABLE`。
- 响应只提供可理解的用户文案和重试建议，不回显微信原始响应、凭据或内部策略细节。

内容安全失败不能写入共享版本，也不能删除或清空草稿。

## 9. 错误与并发语义

| 场景 | HTTP | 处理 |
| --- | --- | --- |
| 表单或聚合不完整 | 422 | 返回字段和所属步骤，保留输入 |
| 内容不合规 | 422 | 保留草稿或编辑稿，不修改共享版本 |
| 图片未审核、停用或不存在 | 422 | 保留草稿，要求重新选图 |
| 菜谱版本冲突 | 409 | 保留本地输入或编辑稿，停止自动保存 |
| 已发布基线版本冲突 | 409 | 保留编辑稿，展示最新共享版本 |
| 权限变化 | 403 | 停止共享操作；个人草稿按所有权规则保留 |
| 内容安全服务不可用 | 503 | 保留草稿，允许用户稍后显式重试 |

聚合版本从持久化新行的 `1` 开始，每次成功写入递增一次。版本冲突、锁等待、死锁或唯一键竞争统一转换为可恢复的业务冲突，不泄漏 SQL 错误。非幂等发布或归档请求不能由客户端自动重放。

## 10. 代码职责边界

后端按职责拆分，不继续扩大现有 `DinnerRecipeService`：

- 菜谱查询服务：家庭列表、详情、可见性和展示映射。
- 草稿编辑服务：创建、分步骤保存、整组替换和版本控制。
- 发布服务：完整性、权限、图片、内容安全、状态转换与编辑稿应用。
- 图片资产查询服务：只返回可用资产及稳定派生地址。
- 内容安全网关：隔离微信实现和测试实现。

前端至少拆分：

- 菜谱与图片 service 契约；
- 纯表单验证和发布完整性计算；
- 自动保存串行队列；
- 家庭菜谱列表页面；
- 分步编辑器页面；
- 图片选择页面。

自动保存队列必须是可独立单测的纯 TypeScript 协调单元，页面只负责收集输入、显示状态和调用协调单元。

## 11. 分阶段交付

### 阶段 A：首个端到端竖切

- V6 菜谱、默认做法、步骤、图片资产和聚合版本模型。
- 家庭列表、个人草稿和分步编辑器。
- 食材、默认做法、真实图片、预览和首次发布。
- 内容安全适配层及失败保留草稿。
- 发布后双方在家庭菜谱列表可见。

### 阶段 B：共同编辑、复制与归档

- 已发布菜谱的个人编辑稿和基线冲突处理。
- 系统菜复制。
- 已发布家庭菜谱归档。
- 创建者、最后修改者和编辑状态展示。

### 阶段 C：做法变体、详情与今晚菜单

- 其他做法版本和默认做法切换。
- 菜谱详情与库存满足状态。
- 加入今晚菜单时保存做法选择。
- 双方做法冲突的显式决策。
- 完成记录保存最终做法、步骤、食材和图片快照。

### 阶段 D：真实图库扩充与正式版收口

- 扩充内部真实图片库并完成逐图许可证据。
- 替换八道系统菜的生成式占位图。
- 完成裁切、派生图、性能和来源展示验收。
- 更新隐私、UGC、内容安全和提审材料。

## 12. 测试与验收

### 12.1 后端

- V6 迁移契约测试，证明不修改 V1–V5。
- 草稿所有权、家庭共享可见性和系统菜只读测试。
- 每个步骤的期望版本、成功递增和冲突保留测试。
- 食材“适量”、默认做法唯一性和有序步骤测试。
- 未选图、非 `APPROVED` 图片、内容拒绝和内容服务不可用测试。
- 首次发布原子性和失败保持 `DRAFT` 测试。
